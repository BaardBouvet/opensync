# Plan: JSON Files Sync POC

**Status:** complete — historical  
**Date:** 2026-04-03  

## Goal

Build a self-contained proof-of-concept that demonstrates the core sync loop using two `jsonfiles`
connector instances as stand-ins for two real systems. No server, no config files — a single
runnable TypeScript script. Running it produces observable output showing records and associations
flowing bi-directionally.

## Scenario

Two systems — **System A** and **System B** — each store `customers` and `orders` as local JSON
files. A minimal in-process sync engine polls both systems and propagates changes, tracking an
identity map so that associations (order → customer) resolve to the correct ID in each system.

### Demo sequence (scripted in the runner)

1. Write a customer (`Alice`) into System A → engine syncs it to System B, printing both IDs.
2. Write an order (`$99`, referencing Alice in System A) into System A → engine syncs to System B,
   resolving the `customerId` association to Alice's System B ID.
3. Edit Alice's name to `Alice Smith` in System A → change flows to System B.
4. Edit the order amount to `$149` in System B → change flows to System A.

## Data Models

Both systems use identical minimal schemas. The `jsonfiles` connector is configured with:

```
idField: "_id"
watermarkField: "_updatedAt"
associationsField: "_associations"
```

### Customer

| Field | Type | Notes |
|-------|------|-------|
| `_id` | string | UUID, generated by connector on insert |
| `name` | string | Mutable — edits should flow |
| `_updatedAt` | string | ISO 8601 — drives incremental reads |

### Order

| Field | Type | Notes |
|-------|------|-------|
| `_id` | string | UUID |
| `amount` | number | Mutable — edits should flow |
| `customerId` | string | Foreign key — must be remapped per system |
| `_updatedAt` | string | ISO 8601 |
| `_associations` | array | `[{ predicate: "customerId", targetEntity: "customers", targetId: "<id>" }]` |

## Engine Design

The POC engine lives entirely inside `sync-poc.ts`. It is intentionally simple — not a reference
implementation of the full engine spec — just enough to prove the sync loop and association remapping
work end-to-end.

### Identity Map

A plain in-memory `Map<string, Map<string, string>>`:

```
identityMap[entityName][`${sourceInstanceId}:${sourceId}`] = targetId
```

When the engine inserts a record into a target and receives a generated ID back, it immediately
stores both directions:

```
identityMap["customers"]["A:alice-a-id"] = "alice-b-id"
identityMap["customers"]["B:alice-b-id"] = "alice-a-id"
```

### Association Remapping

Before inserting or updating a record in a target, the engine rewrites each `Association` entry:

```
targetId = identityMap[assoc.targetEntity][`${sourceInstance}:${assoc.targetId}`]
```

If the target ID is not yet in the map the record is deferred and retried on the next cycle.

### Sync Loop (per cycle)

For each system (A and B):
1. Call `entity.read(ctx, since?)` for each entity.
2. For each record returned:
   a. Remap associations using the identity map.
   b. Check if the record already exists in the other system via the identity map.
   c. If new → call `insert()`, store the returned ID in the identity map.
   d. If existing → call `update()` with only the mutable fields (strip `_id`, `_updatedAt`).
3. Advance the `since` watermark for the next cycle.

No conflict resolution in the POC — last write wins (whichever system polled most recently).
The full engine spec covers proper field-level conflict resolution.

### Echo Prevention

After writing to a target, the engine records the watermark of the write. On the next poll of that
target, records whose `_updatedAt` matches a locally-emitted write are skipped. This prevents A→B
writes from being re-synced back as B→A on the next cycle.

A minimal implementation: after each insert/update, record `emittedIds[instanceId].add(id)` and
skip those IDs on the immediately following read of that instance.

## Implementation Steps

1. **Scaffold directory structure** — create `poc/` and the `data/` subdirectories; add
   `poc/data/` to root `.gitignore`.

2. **Write `sync-poc.ts`** — imports `jsonfiles` connector directly from
   `../connectors/jsonfiles/src/index.js`. No bundling needed (Bun runs TS natively).

3. **Implement the identity map** and association-remapping logic.

4. **Implement the sync loop** — `syncOnce(from, to)` function that reads from one connector
   and writes to the other.

5. **Write the demo script** — a top-level `async function main()` that executes the four scenario
   steps in sequence, calling `syncOnce` between each step and printing a status table to stdout.

6. **Verify** — run `bun run poc/sync-poc.ts` and confirm the output matches the expected scenario.

## Expected Output (sketch)

```
[step 1] Insert customer Alice into System A

[step 2] Insert order $99 (customer: Alice/A) into System A

[step 3] Update customer Alice → Alice Smith in System A
  sync A → B
  customers: A:xxxxxxxx / B:yyyyyyyy (UPDATE name="Alice Smith")

[step 4] Update order amount $99 → $149 in System B
  sync B → A
  orders: B:bbbbbbbb / A:aaaaaaaa (UPDATE amount=149)
```

## Out of Scope for POC

- Persistent identity map (in-memory only; restarting the POC resets it)
- Multi-entity dependency ordering beyond deferred retries
- Conflict resolution (no simultaneous edits to the same field)
- CLI / config files
- Tests (the script is its own integration demo)
